## 2. 简短讲稿

### 为什么做

很多学习管理工具记录的是“学了多久”，但同样 60 分钟可能对应完全不同的效率。本项目希望用手机端低成本记录学习过程，把自报告状态和可选运动特征结合起来，为学习效率预测提供数据基础。

### 做了什么

目前系统已经完成 P0 主链路：手机端 simple-login、开始学习、计时、结束表单、历史记录，以及 FastAPI 后端保存会话和可选运动聚合特征。用户即使没有传感器权限，也能完成学习记录。真实采集时使用用户自选昵称，不收集真实姓名、手机号或学号，并建议多次参与时尽量使用同一个昵称。

### AI 如何体现

AI 部分体现在后续模型流水线：将 `study_sessions` 和 `motion_features` 清洗为特征，按 `efficiency_score` 派生 `low`、`medium`、`high` 标签，再训练 `RandomForestClassifier`，输出 Accuracy、F1-score、混淆矩阵和特征重要性。当前阶段展示采集闭环与训练准备，不展示真实模型准确率，也不声称已经得到可靠模型结论。

### 创新在哪里

创新点不是单一算法，而是把手机轻量感知、自报告学习状态、后端数据闭环和可解释分类模型结合在一个适合大学生自习场景的最小系统中。

### 风险如何解决

传感器兼容性通过降级提示解决；权限拒绝、HTTP/HTTPS 限制、浏览器不支持、后台或锁屏暂停、采样不足都属于 Web 端正常降级，不影响开始学习、结束学习、自报告和历史记录主流程。数据量不足通过先采 20 条、目标至少 50 条、理想 80-120 条逐步缓解；小样本下模型结果只用于课程实验和趋势分析，不能泛化到所有大学生。

### 安全讲稿

- 当前演示只证明系统能形成学习记录采集闭环，并已准备训练前导出和质量检查规则；尚无真实模型实验结果。
- 传感器不可用是手机 Web 端的正常兼容性风险，系统会记录缺失并继续保存自报告数据。
- 后续若样本量不足，模型指标只能作为课程实验的限制性观察，不能写成普遍有效的学习效率预测结论。

- 准备一个测试昵称，不使用真实姓名、学号或手机号。

- 检查左上角返回键和手机系统返回键：学习中不结束会话，结束表单返回前会提示确认。
