# 真实数据采集计划

## 1. 采集目标

本计划用于指导“大学生学习效率预测系统”的真实数据采集。目标是在不收集敏感身份信息的前提下，获得可用于课程项目和科研训练的真实学习记录，为后续三分类模型训练和答辩展示提供基础。

阶段目标：

| 阶段 | 目标 |
| --- | --- |
| 试采集 | 5-10 名参与者，每人 1 条，验证流程可用。 |
| 本周目标 | 至少 20 条真实 completed 记录，覆盖不同地点、任务和时间段。 |
| 展示建议 | 至少 50 条真实有效 completed 记录，再进行课程项目限制性训练。 |
| 理想目标 | 80-120 条真实有效 completed 记录，便于更稳定地划分训练/测试集。 |

低于 30 条真实有效 completed 记录时，只建议做流程验证和数据质量展示，不建议汇报正式模型结论。

## 2. 数据类型边界

| 类型 | 定义 | 处理方式 |
| --- | --- | --- |
| 真实数据 | 参与者真实学习结束后按实际体验提交的记录 | 可进入质量检查和正式训练候选集。 |
| 测试数据 | 调试页面、接口、数据库时产生的临时记录 | 标记并排除，不计入真实样本。 |
| demo/mock 数据 | seed 脚本或人工构造的演示数据 | 只用于演示系统流程，不能写入真实结论。 |

任何报告、PPT 或答辩发言中，都必须把 demo/mock 与真实采集数据分开描述。

## 3. 参与者要求

建议参与者为大学生，优先选择同专业或同年级同学，便于沟通采集流程。参与者不需要安装 App，只需使用手机浏览器访问系统。

参与者填写要求：

- 使用不含真实姓名、学号、手机号、宿舍号或精确定位信息的自选昵称。
- 同一参与者多次记录必须使用同一个昵称。
- 不同参与者不要共用同一个昵称。
- 学习结束后按真实体验填写，不迎合系统，不故意制造高分或低分。

## 4. 采集前准备

| 准备项 | 检查内容 |
| --- | --- |
| 系统入口 | 确认手机端页面可访问，开始学习、结束学习和历史记录可用。 |
| 后端服务 | 确认学习记录和运动特征接口可保存数据。 |
| 昵称规则 | 准备示例昵称，如 `student_a`、`student_b`。 |
| 隐私说明 | 告知不收集真实姓名、学号、手机号、精确定位和通讯录。 |
| 传感器说明 | 告知运动检测可选，拒绝权限或不可用不影响提交。 |
| 数据标记 | 准备记录哪些昵称或时间段属于测试、demo/mock 或真实采集。 |

## 5. 单次采集流程

1. 参与者用手机浏览器打开系统页面。
2. 使用去标识化昵称进入系统。
3. 准备开始真实学习时点击“开始学习”。
4. 学习过程中尽量保持网页在前台，不锁屏，不频繁切换 App。
5. 学习结束后点击“结束学习”。
6. 按本次真实感受填写地点、任务、自报告量表和效率评分。
7. 提交后确认历史记录中出现本次学习记录。
8. 如果传感器不可用或未授权，仍然提交自报告记录。
9. 如果误点或本次学习不应计入有效记录，使用“放弃本次学习”。

## 6. 自报告字段说明

| 字段 | 说明 |
| --- | --- |
| `location` | 本次主要学习地点，如宿舍、图书馆、教室、自习室或其他。 |
| `task_type` | 本次主要任务，如课程作业、考试复习、编程、论文阅读、考研准备或其他。 |
| `goal_clarity` | 1 表示目标很不清楚，5 表示目标非常明确。 |
| `light_level` | 1 表示光照明显影响学习，5 表示光照舒适。 |
| `noise_level` | 1 表示很安静，5 表示噪声严重干扰。 |
| `fatigue_level` | 1 表示不疲劳，5 表示非常疲劳。 |
| `mood_stress` | 1 表示轻松压力小，5 表示压力大或情绪影响明显。 |
| `phone_distraction` | 1 表示几乎未被手机打断，5 表示手机干扰很强。 |
| `efficiency_score` | 本次学习效率主观自评，1-2 为低，3 为中，4-5 为高。 |

## 7. 手机轻量感知说明

项目使用手机浏览器的运动传感能力作为辅助输入，只保存聚合特征：

- `move_count`：检测为移动的次数。
- `shake_count`：明显晃动次数。
- `still_ratio`：检测期间静止比例。
- `avg_acceleration`：加速度幅值平均值。
- `max_acceleration`：加速度幅值最大值。

注意事项：

- 这些特征只能反映手机运动状态，不能直接等同于专注度。
- 传感器权限被拒绝、浏览器不支持、页面后台暂停或上传失败时，运动数据可以缺失。
- 缺失运动数据不阻断学习记录提交。
- 后续训练导出时可用 `motion_available` 标记是否采集到运动特征。

## 8. 数据质量规则

| 问题 | 标记建议 | 处理建议 |
| --- | --- | --- |
| 学习时长少于 5 分钟 | `too_short` | 正式训练默认排除，或只做敏感性分析。 |
| 同一用户短时间重复提交 | `duplicate_candidate` | 人工核查，只保留可信记录。 |
| 明显乱填或机械填同一值 | `careless` | 训练前排除。 |
| 测试或演示记录混入 | `test_data` / `demo` / `mock` | 不计入真实样本，不进入正式训练。 |
| 传感器为空 | `motion_missing` | 不直接排除，记录缺失比例。 |
| 用户放弃学习 | `abandoned` | 不进入有效样本，不解释为低效率。 |

## 9. 7 天执行安排

| 日期 | 目标 | 输出 |
| --- | --- | --- |
| 第 1 天 | 确认系统可访问，完成 2-3 条试采集 | 发现流程问题，区分测试记录。 |
| 第 2 天 | 邀请 5-10 名参与者，讲清楚昵称和隐私规则 | 参与者名单以去标识化昵称记录。 |
| 第 3-4 天 | 收集 8-10 条真实记录 | 覆盖宿舍、图书馆、自习室等地点。 |
| 第 5-6 天 | 再收集 8-10 条真实记录 | 覆盖课程作业、复习、编程、论文阅读等任务。 |
| 第 7 天 | 做初步质量检查和样本统计 | 输出真实、测试、demo/mock 数量和标签分布。 |

## 10. 隐私说明话术

可在采集前向参与者说明：

“本项目用于课程/科研训练，只记录学习时间、自报告状态和可选手机运动聚合特征。请不要使用真实姓名、学号、手机号、宿舍号或包含身份信息的昵称。运动检测是可选的，如果手机不支持或你拒绝权限，仍然可以提交学习记录。数据展示时只使用脱敏样例，不展示能识别个人的信息。”

## 11. 训练前检查清单

正式训练模型前，至少完成以下检查：

- 真实 completed 样本数是多少。
- 测试、demo/mock 数据分别有多少，是否已排除。
- `low`、`medium`、`high` 标签分布是否过度不均衡。
- `motion_features` 缺失比例是多少。
- 少于 5 分钟、重复、乱填、已放弃记录是否已标记。
- 是否已把样本、特征、指标计划和限制写入 `docs/EXPERIMENT_LOG.md`。

## 12. 采集结论表达规范

允许表达：

- “已完成真实数据采集流程设计。”
- “已采集若干真实学习记录，并完成质量检查。”
- “demo/mock 数据仅用于流程演示。”
- “真实模型结果需要在样本量达到条件后再训练和报告。”

不允许表达：

- “demo/mock 数据证明模型有效。”
- “当前模型已经能准确预测所有大学生学习效率。”
- “手机运动特征可以直接判断专注程度。”
- “小样本 Accuracy 可以代表真实泛化能力。”
