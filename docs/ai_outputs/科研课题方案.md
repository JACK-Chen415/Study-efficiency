# 科研课题方案

## 1. 课题名称

基于手机轻量感知与自报告数据的大学生学习效率预测系统设计与实现

## 2. 适用场景

本课题适用于大二物联网专业学生的课程设计、科研训练或答辩展示。课题目标不是构建生产级教育平台，而是在 6 周左右完成一个可运行、可采集、可解释的 MVP 原型，展示物联网感知、Web 全栈开发、数据处理和机器学习建模的完整链路。

## 3. 研究背景

大学生自习过程受地点、时间段、学习任务、目标清晰度、疲劳程度、手机干扰和环境感受等因素影响。传统学习记录通常只记录学习时长，难以反映学习状态差异。本课题尝试使用手机浏览器作为轻量入口，在不增加专用硬件的情况下采集学习会话、自报告量表和可选手机运动聚合特征，为后续学习效率三分类建模提供数据基础。

需要强调的是，本课题中的“学习效率”来自学习结束后的主观自评，不等同于考试成绩、绩点或长期学习能力。

## 4. 研究问题

| 编号 | 研究问题 |
| --- | --- |
| RQ1 | 学习时长、时间段、地点、任务类型和自报告状态能否形成结构化学习效率数据集？ |
| RQ2 | 手机运动聚合特征作为辅助输入，能否补充描述学习过程中的手机移动或静止状态？ |
| RQ3 | 在课程级小样本条件下，如何诚实评估三分类模型的有效性和局限？ |
| RQ4 | 如何在真实采集、测试数据和 demo/mock 数据之间建立清晰边界，避免实验结论混淆？ |

## 5. 研究目标

1. 设计并实现手机端学习记录流程，包括开始学习、结束学习、自报告表单和历史记录。
2. 采集可选手机运动聚合特征，包括 `move_count`、`shake_count`、`still_ratio`、`avg_acceleration` 和 `max_acceleration`。
3. 建立后端 API 和数据库结构，保存用户、学习会话、运动特征和后续预测结果。
4. 完成训练前数据导出和质量检查，区分真实数据、测试数据和 demo/mock 数据。
5. 在真实有效样本达到最低条件后，训练低/中/高学习效率三分类模型，并记录可复现实验日志。
6. 形成可用于答辩的系统架构、数据采集流程、模型流程、风险说明和演示材料。

## 6. 技术路线

```text
手机端学习打卡
-> 结束后填写自报告量表
-> 可选采集 DeviceMotionEvent 聚合特征
-> FastAPI 保存学习会话与运动特征
-> 导出训练 CSV 并进行数据质量检查
-> 满足样本条件后训练 RandomForest 三分类模型
-> 记录 Accuracy、Precision、Recall、F1、混淆矩阵和特征重要性
-> 在看板中展示趋势、预测结果和限制说明
```

## 7. 数据设计

### 7.1 数据来源

| 数据来源 | 定义 | 使用方式 |
| --- | --- | --- |
| 真实数据 | 参与者在真实学习后提交的 completed 记录 | 经过质量检查后可用于正式训练和统计。 |
| 测试数据 | 开发调试接口或页面时提交的记录 | 只用于功能验证，不进入正式实验结论。 |
| demo/mock 数据 | seed 脚本或人工构造的演示数据 | 只用于演示导出、训练流程或页面效果，不得写成真实结论。 |

### 7.2 核心字段

| 类别 | 字段 |
| --- | --- |
| 基础会话 | `start_time`、`end_time`、`duration_minutes`、`time_period` |
| 学习场景 | `location`、`task_type` |
| 自报告状态 | `goal_clarity`、`light_level`、`noise_level`、`fatigue_level`、`mood_stress`、`phone_distraction` |
| 标签 | `efficiency_score`、`efficiency_label` |
| 运动辅助特征 | `move_count`、`shake_count`、`still_ratio`、`avg_acceleration`、`max_acceleration`、`motion_available` |

### 7.3 标签规则

| 自评得分 | 效率标签 |
| --- | --- |
| 1-2 | `low`，低效率 |
| 3 | `medium`，中等效率 |
| 4-5 | `high`，高效率 |

## 8. 方法设计

### 8.1 数据采集方法

- 使用手机浏览器完成学习开始、计时和结束表单。
- 运动数据只保存聚合特征，不保存原始传感器事件。
- 传感器不可用、用户拒绝权限或页面切到后台时，学习记录主流程仍可完成。
- 使用自选昵称区分参与者，不收集真实姓名、学号、手机号、宿舍号或精确定位。

### 8.2 数据质量控制

| 问题 | 处理原则 |
| --- | --- |
| 学习时长少于 5 分钟 | 标记为 `too_short`，正式训练默认排除或单独说明。 |
| 测试或演示记录混入 | 标记为 `test_data`、`mock` 或 `demo`，不计入真实样本。 |
| 明显乱填 | 标记为 `careless`，训练前排除。 |
| 运动特征缺失 | 不直接删除样本，导出时填 0 并加入 `motion_available`。 |
| 已放弃记录 | 不进入有效学习记录和训练数据，也不作为低效率样本。 |

### 8.3 建模方法

计划采用 `RandomForestClassifier` 进行低/中/高三分类。选择该模型的原因是：

- 适合课程项目中的结构化表格数据。
- 能处理数值特征和经过编码的类别特征。
- 可输出特征重要性，便于答辩解释。
- 相比复杂深度模型，更适合小规模原型验证。

正式训练前应先检查真实有效样本数。少于 30 条真实 completed 记录时，不建议训练正式模型；课程展示建议至少 50 条，理想情况为 80-120 条。

### 8.4 对照方案

| 方案 | 目的 |
| --- | --- |
| 多数类 baseline | 检查模型是否只是预测最多的类别。 |
| 自报告 only | 判断自报告字段本身的解释能力。 |
| 自报告 + 运动特征 | 判断运动辅助特征是否带来补充信息。 |
| 自报告 + 运动特征 + `motion_available` | 显式处理传感器缺失机制。 |

## 9. 评价指标

三分类实验应记录：

- Accuracy
- Precision macro 或 weighted
- Recall macro 或 weighted
- F1-score macro 或 weighted
- 混淆矩阵
- 特征重要性

若样本量较低，应把指标表述为“课程项目中的初步结果”或“流程验证结果”，不能写成面向所有大学生的普遍结论。

## 10. 隐私与伦理说明

1. 项目只要求使用自选昵称，不收集真实姓名、学号、手机号、身份证号、宿舍号或精确定位。
2. 手机运动特征为聚合后的轻量统计，不保存原始加速度事件。
3. 自选昵称只能称为去标识化，不能承诺绝对匿名。
4. 报告和 PPT 只展示脱敏样例，不展示可识别个人的信息。
5. 参与者应知道传感器权限是可选的，拒绝权限不影响学习记录提交。

## 11. 预期成果

| 成果 | 说明 |
| --- | --- |
| 可运行系统 | 手机端采集页面、FastAPI 后端和数据库主链路。 |
| 数据规范 | API 文档、数据字典、采集规范和实验日志。 |
| 训练数据 | 真实数据达到条件后导出的训练 CSV 和质量报告。 |
| 模型实验 | 真实训练后记录模型指标、混淆矩阵和限制说明。 |
| 答辩材料 | 架构图、流程图、演示脚本、风险说明和应答库。 |

## 12. 当前不可声称的结论

在真实有效样本不足、模型尚未训练或仅使用 demo/mock 数据时，不能声称：

- 系统已经能准确预测大学生学习效率。
- 手机运动特征已经证明能代表专注度。
- 某个地点、时间段或任务类型一定导致高效率。
- 当前模型指标可以代表真实泛化能力。

更稳妥的表述是：本课题已经建立了面向真实采集的系统原型和实验设计，后续将在真实有效样本积累后进行限制性建模与评估。
