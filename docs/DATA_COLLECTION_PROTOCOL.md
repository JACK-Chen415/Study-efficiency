# 真实数据采集规范

> 文档状态：Milestone 3C 采集前规范  
> 目的：在不收集敏感身份信息的前提下，指导参与者使用自选昵称完成去标识化数据记录。

## 1. 采集目的

本项目需要收集大学生日常自习场景下的学习记录，包括学习起止时间、自报告状态、效率评分，以及可选手机运动聚合特征。数据用于后续训练低/中/高学习效率三分类模型，并分析哪些因素可能与效率评分相关。

本规范只描述采集流程和质量控制，不得把尚未采集或模拟生成的数据写成真实实验结论。

## 2. 数据来源标记

| 类型 | 定义 | 使用限制 |
| --- | --- | --- |
| 真实数据 | 参与者在真实学习后提交的记录 | 可进入正式数据清洗和模型训练，但需使用自选昵称或去标识化昵称区分用户并完成质量检查。 |
| 测试数据 | 开发或演示时为了检查接口临时提交的记录 | 只能用于功能验证，不进入真实实验统计。 |
| 模拟数据 | 为演示、seed 或模型流程占位人工构造的数据 | 必须明确标注为模拟，不得写成真实采集结果。 |

## 3. 参与者一次记录流程

1. 使用手机浏览器打开前端页面：`http://106.55.60.243/study/`。
2. 使用不含真实姓名、学号、手机号、宿舍号或精确定位信息的自选昵称登录，例如 `student_a`。同一参与者多次记录必须使用同一个昵称，不同参与者不要共用昵称。
3. 在准备开始一段学习时点击“开始学习”。
4. 正常学习，尽量保持网页在前台打开，不要锁屏、不要划掉浏览器进程，也不要频繁切换到其他 App。
5. 学习结束后点击“结束学习”。
6. 按本次真实体验填写自报告字段，不按期望结果迎合系统。
7. 提交后确认历史记录中出现本次学习记录。
8. 如果手机运动检测不可用，可以直接提交；该记录的 `motion_features` 允许为空。
9. 如果开始后发现误点、临时取消或本次学习不应计入有效记录，使用“放弃本次学习”。放弃记录会保留为 `abandoned` 状态，不进入有效学习记录或训练数据。

## 4. 自报告字段填写解释

量表方向需在采集前向参与者说明清楚：`goal_clarity`、`light_level` 高分偏正向，表示目标更明确或光照更舒适；`noise_level`、`fatigue_level`、`mood_stress`、`phone_distraction` 高分偏负向，表示干扰、疲劳、压力或手机打断更强。`efficiency_score` 是本次学习的主观总体自评，不等同于客观成绩。

| 字段 | 填写方式 | 解释 |
| --- | --- | --- |
| `location` | 选择 `dormitory`、`library`、`classroom`、`study_room` 或 `other` | 选择本次主要学习地点。如果地点混合，选择停留时间最长的地点。 |
| `task_type` | 选择 `coursework`、`exam_review`、`coding`、`paper_reading`、`postgraduate_prep` 或 `other` | 选择本次主要学习任务。如果同时做多件事，选择耗时最多或最重要的一项。 |
| `goal_clarity` | 1-5 分 | 1 表示开始前几乎不知道要完成什么；5 表示目标非常明确且可检查。 |
| `light_level` | 1-5 分 | 1 表示光线很差或刺眼影响学习；3 表示基本可接受；5 表示光照舒适。 |
| `noise_level` | 1-5 分 | 1 表示非常安静；5 表示噪声明显干扰学习。填写时按干扰程度，不按客观分贝。 |
| `fatigue_level` | 1-5 分 | 1 表示不疲劳；5 表示非常疲劳、困倦或难以集中。 |
| `mood_stress` | 1-5 分 | 1 表示心情轻松、压力小；5 表示压力很大或情绪明显影响学习。 |
| `phone_distraction` | 1-5 分 | 1 表示几乎没有被手机打断；5 表示频繁刷手机或被消息严重打断。 |
| `efficiency_score` | 1-5 分 | 对本次学习效率的总体自评。1-2 将转为低效率，3 为中等效率，4-5 为高效率。 |

## 5. 采集目标

- 最低目标：至少 50 条真实有效记录，才能开始做初步三分类训练和限制性说明。
- 理想目标：80-120 条真实有效记录，便于更稳定地划分训练/测试集并观察特征趋势。
- 本周先收集 20 条：建议 5-10 名参与者，每人提交 2-4 条，覆盖不同地点、任务和时间段。

## 6. 本周 20 条执行建议

| 日期安排 | 目标 | 说明 |
| --- | --- | --- |
| 第 1 天 | 确认参与者和演示流程 | 说明自选昵称规则，统一使用测试外的真实采集昵称。 |
| 第 2-3 天 | 收集 8-10 条 | 优先覆盖图书馆、宿舍、自习室等常见地点。 |
| 第 4-5 天 | 再收集 8-10 条 | 尽量补齐课程作业、复习、编程、论文阅读等任务类型。 |
| 第 6 天 | 初步质量检查 | 标记过短、重复、明显乱填或测试记录。 |
| 第 7 天 | 汇总缺口 | 判断是否继续采集，准备模型训练前的数据导出需求。 |

## 7. 昵称使用与隐私原则

- 不收集真实姓名、学号、手机号、身份证号、宿舍号、精确定位或通讯录信息。
- 昵称由用户自行选择，仅用于系统区分用户，建议使用 `student_a` 这类不包含真实身份信息的去标识化昵称。
- 同一参与者多次记录时必须使用同一个昵称；不同参与者应避免共用同一个昵称。
- 远端服务可能存在服务器访问日志，因此本项目更准确的表述是“去标识化采集”，不应宣称为绝对匿名采集。
- 报告和 PPT 只展示脱敏样例，不展示能够识别个人的信息。
- 原始数据不得上传到无关外部服务或公开仓库。

## 8. 传感器与缺失数据

- 手机运动检测是辅助特征，不是提交学习记录的必要条件。
- 传感器不可用的常见原因包括：用户拒绝权限、HTTP/HTTPS 限制、浏览器不支持、页面切到后台或锁屏导致暂停、采样时长不足或上传失败。
- 出现上述情况时，仍允许提交自报告记录，不得阻断学习记录主流程。
- 系统会尝试使用浏览器 Screen Wake Lock 减少锁屏导致的采集中断；不支持或启用失败时只给出提示，不影响记录保存。
- 纯 Web 页面无法保证切到后台、锁屏或关闭浏览器后仍持续采集运动传感器。页面会提示后台期间传感器可能暂停，并在前端显示后台暂停次数和时长。
- 对应的 `motion_features` 可以为空；后续特征工程需在 `EXPERIMENT_LOG.md` 中说明缺失处理方法。

## 9. 数据质量规则

| 问题 | 标记建议 | 处理建议 |
| --- | --- | --- |
| 学习时长过短 | 少于 5 分钟的 completed 记录标记为 `too_short` | 功能测试可保留；正式训练默认排除，或仅用于敏感性分析。 |
| 重复提交 | 同一用户短时间内内容高度相同标记为 `duplicate_candidate` | 只保留最可信的一条，其他不进入训练。 |
| 明显乱填 | 所有量表机械填同一值、效率评分与描述严重矛盾、任务地点明显不合理 | 标记为 `careless`，训练前排除。 |
| 测试或演示记录混入 | 昵称、时间或备注显示为测试用途 | 标记为 `test_data`，不得计入真实样本数。 |
| 传感器为空 | `motion_features` 缺失 | 不直接排除；模型阶段按缺失策略处理。 |
| 已放弃学习 | `abandoned_at` 不为空 | 只作为流程质量记录或异常中断记录；不计入有效样本数，不进入训练数据，也不作为低效率样本。 |

## 10. 进入模型训练前检查

- 真实有效记录数、测试数据数和模拟数据数分别统计。
- 检查 `efficiency_label` 的 `low`、`medium`、`high` 分布，避免只有单一类别。
- 确认关键自报告字段没有大量缺失或明显异常。
- 明确运动特征缺失比例和后续填补策略。
- 统计 `abandoned`、`too_short`、测试数据和模拟数据数量，说明排除原因。
- 将样本量、纳入字段和剔除规则写入 `docs/EXPERIMENT_LOG.md`。
